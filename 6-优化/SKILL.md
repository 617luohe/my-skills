# 6-优化 — 重构与架构改进

两个模式：**架构分析**（发现问题）和 **重构计划**（解决问题）。

## 模式路由（进入时自动判断）

根据用户输入和上下文自动选择模式：

**架构分析模式信号**（模式 A）：
- ✅ 用户说"看看有什么问题"、"分析一下"、"代码很乱"但没有明确重构目标
- ✅ 上游来自 7-调试，发现架构债务（测试接缝缺失、耦合过深）
- ✅ 代码审查（5-检查）发现架构摩擦

**重构计划模式信号**（模式 B）：
- ✅ 用户说"重构 X 模块"、"拆分 Y 类"、"提取 Z"等明确动作
- ✅ 架构分析（模式 A）完成后，用户选定候选方案
- ✅ 上游来自 1-规划，PRD 包含重构任务

**如果信号不明确**：

   - **重构计划**：针对明确目标，产出极小提交清单（产出可执行计划 + 后续进 4-开发）

## MUST 规则

1. **先探索后判断。** 走读代码前先读 CONTEXT.md + ADR。
2. **候选方案用领域术语 + 架构术语。** CONTEXT.md 词汇 + 接缝/深度/浅模块。
3. **重构计划每个提交必须能独立工作。** 每步尽可能小，始终可回滚。
4. **ADR 不轻启。** 除非确实值得重开，才标记冲突。

## 模式 A — 架构分析

发现代码中的架构摩擦，提出深化机会——把浅模块变深的改进方案。

### 核心概念

| 术语 | 定义 |
|---|---|
| **模块** | 任何有接口与实现的单元（函数、类、包、切片） |
| **接口** | 调用方必须知道的一切：类型、约束、错误模式、调用顺序、配置 |
| **实现** | 模块内部代码 |
| **深度** | 小接口封装大逻辑 = 深模块（好） |
| **浅模块** | 接口几乎和实现一样复杂（坏） |
| **接缝** | 接口所在的位置，可在不修改原处的情况下改变行为 |
| **适配器** | 在接缝处对接口的一个具体实现 |

规则补充：
- 一个接缝只有 1 个适配器时，通常只是"假想接缝"；出现 2 个及以上适配器才是"真实接缝"
- **接口是测试面**：优先通过接口测试行为，避免把复杂度泄露给调用方
| **删除测试** | 删掉这个模块：复杂度消失还是散落到 N 个调用方？ |

### 步骤

1. **探索** — 读 CONTEXT.md + ADR，走读代码，注意：
   - 理解一个概念需要跳转很多小模块
   - 接口几乎和实现一样复杂（浅）
   - 代码部分没有测试，或现有接口很难测
   - 应用**删除测试**：删掉这个模块，复杂度消失还是散落到 N 个调用方？
2. **展示候选** — 对每个候选展示：文件、问题、方案、收益。使用领域术语（用 CONTEXT.md 词汇），架构术语用规范词汇（接缝、深度、浅模块）
3. **追问循环** — 你选定一个后深入讨论

**ADR 冲突处理**：如果候选方案与已有 ADR 矛盾，只有当你认为这个 ADR 确实值得重新审视时才提出来。标记清楚（如"与 ADR-0007 冲突，但值得重开因为……"）。不要列出每个被 ADR 禁止的理论方案。

## 模式 B — 重构计划

通过采访了解重构需求，输出详细的渐进式方案。

### 步骤

1. **听取问题描述** — 你说要解决什么问题
2. **探索代码库** — 验证描述，了解现状
3. **检查测试覆盖** — 这片代码有测试吗？覆盖够吗？如果不够，问你测试计划
4. **探讨其他选项** — 给你展示其他方案
5. **敲定范围** — 明确要改什么、不改什么
6. **拆解为极小提交** — 每一步尽可能小，始终可工作

### Issue 模板

```md
## Problem Statement
[开发者面临的问题]

## Solution
[解决方案]

## Commits
1. [最小提交] 步骤 1 → 可工作
2. [最小提交] 步骤 2 → 可工作
...

## Decision Document
已决策的模块划分、接口变更、技术选型、架构决策等。
不包含具体文件路径（会过期）。

## Testing Decisions
- 哪些模块需要测试
- 测试的依据（只测外部行为，不测实现细节）
- 代码库中类似的测试参考

## Out of Scope
[本次不涵盖的内容]
```

---

## 什么时候用

- **模式 A**：代码越来越乱需要系统性梳理、测试很难写怀疑是架构问题
- **模式 B**：模块变得臃肿需要重构、想确保每步可回滚

## 案例

```
你：payment_service.py 800 行了，帮我优化
Claude：我看看... 确实，支付处理、退款、对账、通知四个功能耦合在一起。

       建议按以下步骤重构：
       1. [最小] 提取 PaymentStatus 常量和枚举 → 不影响逻辑
       2. [最小] 提取 PaymentProcessor 类（仅支付逻辑）
       3. [最小] 提取 RefundProcessor 类
       4. [最小] 提取 ReconciliationService 类
       5. [最小] 提取 NotificationService 类

       每一步独立可提交，任意步骤停下都能正常工作。
       你觉得粒度合适吗？
```

## 完成后 — 自动推进

重构计划敲定后进入执行，一句话报出下一步再执行：

- 计划确认、开始按提交清单实现 → 转 `/4-开发`（每步 TDD + 全绿再下一步）。
- 每完成一批可提交的改动 → `/8-版本管理` 落盘，保持每步可回滚。
- 架构分析只出结论、暂不动手 → 停在这里，等用户决定何时开工。
