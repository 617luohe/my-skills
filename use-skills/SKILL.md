# use-skills — 智能技能调度器

输入你的需求，我自动为你调用最合适的一个或多个技能。你不需要记住哪个技能叫什么名字。

## MUST 规则

1. **匹配到技能后必须先展示计划再执行。** 格式：`→ 分析需求 → 匹配技能 → 开始执行`
2. **多技能必须按顺序串联执行，不并行。** 每个技能执行完毕后汇总结果再进入下一个。
3. **严格遵循目标技能 SKILL.md 的全部指令。** 包括其 MUST 规则、阶段流程、输出格式。
4. **无法匹配任何技能时直接说明，不强行匹配。**

## 工作方式

### 步骤 0 — 任务规模判断（预处理）

在匹配技能前，先判断任务规模，决定执行策略：

**简单任务信号**（直接路由到单个 skill）：
- ✅ 单文件小改动（"修改 config.py 的超时时间"、"给 User 类加个 email 字段"）
- ✅ 已定位的 bug（"修复 auth.py:45 的空指针"、"改正拼写错误"）
- ✅ 明确的配置调整（"把日志级别改成 DEBUG"）

**中等任务信号**（需要工作流串联）：
- ✅ 新功能开发（"加个用户认证功能"、"实现订单取消"）
- ✅ 方案不确定（"优化性能"、"重构代码"需要先分析）
- ✅ 多模块联动（"订单和支付打通"）
- ✅ 需要完整验证（明确说要"测试"、"审查"、"确保质量"）

**复杂任务信号**（转 auto-iteration）：
- ✅ 明确多轮迭代（"自动帮我"、"端到端完成"、"持续改进"）
- ✅ 需要门禁反馈（"提升测试覆盖率到 90%"、"修复所有 mypy 报错"）
- ✅ 涉及多个独立任务（"给 5 个模块都加上日志"）

**Agent 统筹信号**（优先于普通复杂任务判断）：
- ✅ 明确调用 Codex、Cursor 等外部编程智能体
- ✅ 要求多智能体、跨模型协作或并行评审
- ✅ 要求把独立任务委派给外部 agent

**判断结果与执行策略**：
- **简单** → 执行步骤 1-4（当前标准流程）
- **中等** → 规划后通过执行授权门禁，再串联后续工作流（见"组合规则"）
- **复杂** → 一句话说明理由后，移交 `/0--auto-iteration` 接管
- **多 Agent 协作** → 移交 `/0--Agent统筹`；若同时要求持续闭环，由 `/0--auto-iteration` 主控并按轮调用

**判断后验证机制**：

如果执行过程中发现判断错误：
- **简单任务实际复杂**（如"加缓存"涉及架构改动）→ 主动升级路径：
  - 当前进度：已完成 X
  - 升级理由：发现涉及 Y 个模块 + Z 个决策点
  - 建议路径：转 `/1-规划` 重新审视全局，或转 `/0--auto-iteration` 接管剩余部分
  - 询问用户确认升级或继续当前路径
  
- **中等任务实际简单**（如"实现登录"只需调用现有 OAuth 库）→ 简化后续环节：
  - 跳过不必要的审查环节（如已有充分测试覆盖，5-检查可轻量化）
  - 在推进时说明"任务比预期简单，简化后续流程"

**边界案例处理**：

| 用户表述 | 模糊点 | 判断策略 |
|---|---|---|
| "给订单模块加缓存" | 是装饰器(简单)还是分布式缓存(中等)？ | 追问："只加内存缓存还是需要 Redis？" 根据回答判断 |
| "优化性能" | 范围不明 | 默认中等（需分析），追问"哪个模块/API 的性能？" |
| "修复 bug" | 是否已定位？ | 追问："bug 在哪个文件/函数？" 已定位→简单，未定位→中等 |
| "重构代码" | 规模不明 | 追问："哪个模块？代码规模多大？" <200 行→简单，>500 行→中等 |

### 步骤 1 — 理解需求

分析用户输入的自然语言，提取核心诉求。识别出：
- **主要意图** — 用户最想做什么
- **次要意图** — 是否有连带需求
- **目标对象** — 针对什么代码/项目

### 步骤 2 — 匹配技能

对照路由表找到匹配的技能。如果有多个匹配，按依赖关系排好顺序。

### 步骤 3 — 展示计划

向用户简要说明匹配结果，确认后再执行。格式：

```
→ 分析需求：<你理解的需求>
→ 匹配技能：<技能1> → <技能2>（如有）
→ 开始执行...
```

### 步骤 4 — 执行技能

读取匹配技能的 `SKILL.md`，严格按照其指令执行。效果等同于直接调用了该技能。

## 组合规则

当需求涉及多个技能时，按下述顺序排列：

### 工作流串联（中等任务）

**核心原则**：按开发阶段自动衔接，形成完整链路。

| 场景 | 默认路径（规划完成后必须等执行授权） |
|---|---|
| 新功能开发 | 1-规划 → [用户确认 PRD/任务清单可执行] →（存疑用 3-原型）→ 4-开发 → 5-检查 → 8-版本管理 |
| 陌生代码改动 | 2-分析 → 1-规划 → [执行授权] → 4-开发 → 5-检查 |
| 重构优化 | 2-分析 → 6-优化 → [用户确认执行计划] → 4-开发 → 5-检查 |
| Bug 修复 | 7-调试 → 4-开发(补回归测试) → 8-版本管理 |
| 阶段收尾 | (当前 skill) → 9-最后整理 |

**串联执行规则**：
1. 按顺序依次执行每个技能
2. 每个技能执行完毕后，简短汇总结果再进入下一个
3. 全部完成后给出一份整体总结
4. 不要同时执行两个技能，必须按顺序串联

### 独立技能路由（简单任务）

1. **分析先行** — 如果包含"分析"意图，先做 2-分析
2. **规划先行** — 如果包含"规划"意图，先做 1-规划
3. **原型先行** — 如果包含"验证"意图，在规划后、开发前做 3-原型
4. **开发/重构** — 中间执行
5. **检查收尾** — 最后做 5-检查、9-最后整理
6. **tools 类技能** — 独立使用，或作为开发流程的输入/输出环节
7. **自动迭代优先** — 如果包含"自动帮我"、"持续改进"、"迭代执行"等多轮闭环意图，直接用 自动迭代 接管整个流程
8. **Agent 统筹意图优先识别** — 明确要求 Codex、Cursor、跨模型或多个外部 Agent 时使用 0--Agent统筹；同时要求多轮闭环时由 0--auto-iteration 主控，按需调用

示例：
- "分析一下这段代码，然后帮我重构" → 2-分析 → 6-优化
- "帮我验证一下这个数据模型，没问题就用 TDD 实现" → 3-原型 → 4-开发
- "帮我规划一个用户认证功能，然后实现它" → 1-规划 → [用户确认 PRD/任务清单可执行] → 4-开发 → 5-检查 → 8-版本管理
- "看看这个项目的问题，做个优化方案" → 2-分析 → 6-优化
- "调研一下竞品，然后做个PPT汇报" → tools--深度研报生成 → tools--幻灯片生成
- "先搜一下最新的技术方案，画个架构图" → tools--智能搜索 → tools--图表生成
- "分析这个模块的代码结构，画个流程图" → 2-分析 → tools--图表生成
- "自动帮我完成缓存层的开发、测试和文档" → 自动迭代（接管完整五阶段闭环）
- "初始化老油条，然后看看这次交什么" → 老油条（初始化） → 老油条（分析建议）
- "规划一个新功能，用老油条管理交付节奏" → 1-规划 → 老油条（初始化）
- "给系统加个缓存功能" → 判断为中等任务 → 1-规划 → [用户确认 PRD/任务清单可执行] → 4-开发 → 5-检查 → 8-版本管理
- "修改 config.py 的超时时间" → 判断为简单任务 → 4-开发
- "自动帮我提升测试覆盖率到 90%" → 判断为复杂任务 → 0--auto-iteration

## 执行策略

### 单一技能
直接读取该技能的 `SKILL.md`，按照其全部指令执行。包括该技能内部的阶段流程、提问方式、输出格式。

### 多技能组合
按顺序依次执行每个技能。每个技能执行完毕后，简短汇总结果再进入下一个。全部完成后给出一份整体总结。

## 注意事项

- 不要同时执行两个技能，必须按顺序串联
- 每个技能执行时，严格遵循其 SKILL.md 中的规则（包括 `disable-model-invocation` 标记）
- 如果用户的需求无法匹配任何技能，直接说明并询问是否需要其他帮助
