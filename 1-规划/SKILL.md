# 1-规划 — 方案设计与任务拆解

写代码前的完整规划流程。按阶段推进，不跳跃。

## 流程

这个技能分 5 个阶段，按顺序执行。在每个阶段结束时问用户是否要继续下一阶段。

### 阶段 1 — 方案追问

把你的方案想象成一棵**决策树**。每个不确定的点是一个分支节点，每个分支走到叶子（做出决策）才算完成。

**核心原则：RELENTLESS** — 沿着依赖链逐个决议，不把所有分支走完不罢休。

**规则**：
- 一次只问一个问题，等你回答后再问下一个
- 每个问题先给出**推荐答案**，再让你选择
- 能读代码解决的**不问你** — 去代码库里找答案
- 沿着决策树的依赖链逐个决议，不做跳跃
- 当前面的决策会影响后面的选择时，先把前面定下来再推进
- **不要因为后面还有阶段就急着收尾** — 阶段1的唯一退出条件是：所有决策分支都已决议，双方达成共享理解

**信号表明可以退出阶段1**：
- [ ] 所有涉及的需求假设都已显式化并确认
- [ ] 所有不确定的技术选择都已做出决策
- [ ] 所有决策之间的依赖关系已理清
- [ ] 双方对"要做什么"没有分歧

**输出**：达成共识的方案描述。

---

### 阶段 2 — 领域术语提取

从对话中提取领域相关的名词和概念，整理成一致的术语表。

**输出**：`UBIQUITOUS_LANGUAGE.md`，包含：
- 规范术语表格（每个术语一句话定义）
- 术语之间的关系（如"一个 Order 产生一个或多个 Invoice"）
- 已标记的歧义项（同一个词指不同东西）
- 示例对话展示术语的真实用法

---

### 阶段 3 — 接口设计

用并行子代理同时生成多种截然不同的设计方案，然后对比。

启动 3+ 个并行 Agent，每个分配不同的设计约束：

| 代理 | 约束方向 |
|---|---|
| Agent 1 | 最小接口 — 目标 1-3 个方法 |
| Agent 2 | 最大灵活性 — 支持多种使用场景 |
| Agent 3 | 针对最常见场景优化 |

每个方案展示：接口签名 + 使用示例 + 隐藏了什么 + 取舍。

**输出**：选定的接口设计方案（可能来自多个方案的组合）。

---

### 阶段 4 — 输出 PRD

基于已讨论的内容产出结构化 PRD。

**不追问用户** — 只基于阶段 1-3 已达成的共识综合整理。

模板：

```
## Problem Statement
用户面临的问题。

## Solution
解决方案概述。

## User Stories
- As a <角色>, I want <功能>, so that <价值>

## Implementation Decisions
已决策的模块划分、接口、技术选型。

## Testing Decisions
哪些模块要测试、怎么测。

## Out of Scope
本次不涵盖的内容。
```

**输出**：完整的 PRD 文档。

---

### 阶段 5 — 拆解任务

将 PRD 拆解为可独立执行的垂直切片，按依赖顺序排列。

每个 issue 是一刀切穿所有层的薄切片（schema → 逻辑 → 测试），不是切一层。

两种类型：
- **AFK** — 可直接实现并合并
- **HITL** — 需要你做决策

**输出**：按依赖顺序排列的 issue 清单，标注类型和前置依赖。

---

## 什么时候用

- 开始一个中型以上功能前
- 方案不确定，需要系统性地审视
- 需要一份正式的 PRD 和任务拆解
- 团队协作时需要分派独立任务

## 案例

```
你：我想给系统加一个缓存层，帮我规划一下
Claude：好的，先从最基本的开始——你要缓存什么数据？
       我建议缓存数据库查询结果，而不是计算中间结果。
       你觉得呢？
你：对，查库结果
Claude：缓存粒度呢？
       A) 全表缓存
       B) 逐行缓存
       C) 按查询参数缓存
       我推荐 C，命中率最高。
你：选 C
...

[进入阶段 2 → 阶段 3 → ... → 阶段 5]

Claude：规划完成，共拆出 4 个 issue：
       1. [AFK] 缓存基础设施 — 基础抽象 + TTL 策略
       2. [AFK] 查询缓存接入 — 缓存装饰器实现
       3. [AFK] 失效策略 — 写入时缓存失效
       4. [HITL] 缓存监控 — 命中率指标 + 告警阈值
```
