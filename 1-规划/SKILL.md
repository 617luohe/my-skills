# 1-规划 — 方案设计与任务拆解

写代码前的完整规划流程。按阶段推进，不跳跃。

## MUST 规则

1. **按顺序走阶段，不跳跃。** 需要用户决策或授权的阶段必须等待；明确规定不追问的综合整理阶段直接执行。
2. **事实自行查证，决策交给用户。** 代码、配置、文档和工具中有唯一答案的是事实；目标、偏好、范围、风险和取舍是决策。调查结果可以支持推荐，不能替用户决策。
3. **每个决策先给推荐答案和理由，再让用户选择。** 不做开放题。
4. **默认批量 grilling。** 每轮集中询问当前 frontier 中主题相近、互不依赖的决策；有依赖的问题不得同轮。
5. **用户要求逐步完善时一次一问。** 出现“一步一步”“逐步完善”“一个一个问”“step by step”“one at a time”等表达后，每次只问一个决策并等待回答。
6. **阶段 1 必须由用户显式确认退出。** frontier 为空后先输出共享理解摘要；用户确认前不得进入阶段 2 或生成规划产物。
7. **理解确认不等于开发授权。** 进入原型或开发前，用户还需确认 PRD 和任务清单可执行。
8. **阶段 4 不追问用户。** 只基于阶段 1-3 已达成的共识综合整理。

## 阶段速查

| 阶段 | 内容 | 核心产物 | 追问用户？ |
|------|------|----------|-----------|
| 1 | 方案追问（文档发现 → 决策树 frontier） | 用户确认的共识方案 | 是（默认批量；可切逐步） |
| 2 | 领域术语提取 | CONTEXT.md | 是（术语确认） |
| 2.5 | ADR 决策记录（可选） | ADR 或 DECISIONS.md | 否 |
| 3 | 接口设计（3 并行 Agent） | 选定接口方案 | 是（方案选择） |
| 4 | 输出 PRD | 结构化 PRD 文档 | 否 |
| 5 | 拆解任务 | Issue 清单（AFK/HITL） | 否 |

### 阶段 1 — 方案追问

把你的方案想象成一棵**决策树**。每个不确定的点是一个分支节点，每个分支走到叶子（做出决策）才算完成。

**核心原则：RELENTLESS** — 沿着依赖链逐个决议，不把所有分支走完不罢休。

#### 1a. 文档发现（进入追问前）

在开始追问之前，先扫描项目中已有的领域文档：

```
grep 项目根目录和 docs/ 查找：CONTEXT-MAP.md、CONTEXT.md、docs/adr/NNNN-*.md
```

- 如果 `CONTEXT-MAP.md` 存在 → 按 map 读取每个上下文的 `CONTEXT.md` 与对应 `docs/adr/`
- 如果 `CONTEXT.md` 存在 → 读取，了解已有的术语体系和概念定义
- 如果 `docs/adr/` 存在 → 读取最近的 ADR，了解历史架构决策
- 如果用户输入中使用的术语与现有文档冲突 → 立即指出："你的 CONTEXT.md 定义 X 为 A，但你刚才说的是 B——哪个是对的？"
- 如果**不存在** CONTEXT.md → 在阶段 2 中按需创建（懒创建：有第一条确认术语时再落盘）

> 这是 grill-with-docs 的核心特性：**带着对现有领域的理解来追问**，而不是从零开始。

**缓存规则**：阶段 1a 读取的文档内容在整个规划过程中复用，不重复读取。如果阶段 2 需要更新 CONTEXT.md，基于阶段 1a 的缓存内容进行增量修改。

**输出**：项目领域文档现状综述（什么存在、什么缺失、有没有术语冲突）。

#### 1b. 决策树 grilling

把未决事项建模为依赖图：节点是决策，边是依赖，**frontier** 是前置决策已解决、当前可以回答的决策集合。

先分类，再处理：
- **事实（Fact）** — 代码、配置、现有文档或工具中存在可验证答案。自行调查，不反问用户。
- **决策（Decision）** — 涉及目标、偏好、范围、风险接受度或真实取舍。必须由用户决定；先给推荐答案和理由。

**默认：批量模式**
- 每轮取当前 frontier，把主题相近且互不依赖的决策集中询问
- 问题编号；每题包含推荐答案、简短理由和可选项
- 一个答案会改变另一个问题前提时，不得放在同一轮
- 高影响、难逆转的决策可以单独成轮
- 某项事实调查只阻塞依赖它的分支，其他独立 frontier 继续推进

**逐步模式**
- 用户说“一步一步”“逐步完善”“一个一个问”“step by step”“one at a time”或同义表达时启用
- 一次只问一个决策，等待回答后再问下一个
- 除非用户明确要求恢复批量，否则本次规划持续使用逐步模式

**所有模式共同规则**：
- 沿依赖关系推进，不越过未解决的前置决策
- 不因后面还有阶段就急着收尾；frontier 为空才表示分支已穷尽
- **场景压力测试** — 用具体场景探测领域边界：「一个已取消的订单还能退款吗？如果退款完成后再取消呢？」
- **代码交叉验证** — 用户描述系统现状时，用实际代码核实；发现冲突就明确指出

**阶段 1 退出门禁**：
1. [ ] 所有需求假设已显式化并确认
2. [ ] 所有技术和产品决策已完成，frontier 为空
3. [ ] 决策依赖关系已理清
4. [ ] 输出共享理解摘要：目标、已验证事实、关键决策、范围外事项、残余风险
5. [ ] 用户显式确认摘要正确，并同意进入阶段 2

若用户指出遗漏，重新加入决策树，不进入后续阶段。确认前只允许只读事实调查；不写 `CONTEXT.md`、ADR、PRD 或任务清单，不进入原型或开发。

**输出**：用户显式确认的共识方案。

### 阶段 2 — 领域术语提取

从对话中提取领域相关的名词和概念，整理成一致的术语表，并维护唯一领域文档 `CONTEXT.md`。只有阶段 1 的共享理解获得用户显式确认后，才能开始写入。

**实时规则**（贯穿整个阶段）：
- **术语冲突标记** — 如果你用词与已有 CONTEXT.md 或之前的讨论冲突，立即指出：「你的术语表定义 X 为 A，但你刚才说的是 B——到底是哪个？」
- **模糊语言锐化** — 当你使用模糊或过载词汇时，提议精确的规范术语：「你说的"account"——是指 Customer 还是 User？这是两个不同的概念。」
- **场景压力测试** — 当领域关系正在定义时，用具体场景探测边界：「如果一个 Customer 下了一个 Order，取消后重新下单，这两个 Order 有关联吗？」
- **内联更新** — 每当一个术语被确认，立即更新 CONTEXT.md。不批量等最后。写一张表就够了。

**输出**：`CONTEXT.md` — 项目唯一领域语言文档，包含规范术语、关系、示例对话和已标记歧义。

> 如果项目已有 CONTEXT.md，在其基础上更新（合并新术语、修改歧义项、补充关系）。
> 如果项目没有 CONTEXT.md，这里按需创建。`CONTEXT.md` 只含术语和领域概念，不含实现细节。

**CONTEXT.md 格式**：
```
# {项目名称}

{一两句描述}

## Language

**Order**:
{A concise description}
_Avoid_: Purchase, transaction

**Invoice**:
A request for payment sent to a customer after delivery.
_Avoid_: Bill, payment request

## Relationships

- An **Order** produces one or more **Invoices**

## Example dialogue

> **Dev:** "When a Customer places an Order, do we create the Invoice immediately?"

## Flagged ambiguities

- "account" was used to mean both Customer and User — resolved: these are distinct.
```

### 阶段 2.5 — 决策记录（ADR）

在术语体系建立后、接口设计前，回顾整个规划阶段已做出的关键决策。

**不所有决策都记录**。仅当以下**三个条件全部满足**时才提议创建 ADR：
1. **难逆转** — 事后改主意的成本很高
2. **少见** — 未来的读者看到代码会想"为什么这样做？"
3. **真实取舍** — 确实有多种选择，你基于特定原因选了一个

如果三条缺任何一条，跳过 ADR。

**ADR 格式**（极简，一条就够了）：
```md
# {短标题}

{1-3 句：上下文是什么、决定了什么、为什么。}
```

**可选扩展**（只有确实有价值时加）：
- **Status** — `proposed | accepted | deprecated | superseded by ADR-NNNN`
- **Considered Options** — 被拒绝的替代方案值得记住时才写
- **Consequences** — 只有非显而易见的后续影响需要指出时才写

**存储位置**：
- 如果项目已有 `docs/adr/`，顺序编号 `NNNN-slug.md`
- 如果项目没有，输出到 `DECISIONS.md` 放在项目根

### 阶段 3 — 接口设计

用并行子代理同时生成多种截然不同的设计方案，然后对比。

启动 3+ 个并行 Agent，每个分配不同的设计约束：

| 代理 | 约束方向 |
|---|---|
| Agent 1 | 最小接口 — 目标 1-3 个方法 |
| Agent 2 | 最大灵活性 — 支持多种使用场景 |
| Agent 3 | 针对最常见场景优化 |

每个方案展示：接口签名 + 使用示例 + 隐藏了什么 + 取舍。

**输出**：选定的接口设计方案（可能来自多个方案的组合）。

---

### 阶段 4 — 输出 PRD

基于已讨论的内容产出结构化 PRD。

**不追问用户** — 只基于阶段 1-3 已达成的共识综合整理。

模板：

```
## Problem Statement
用户面临的问题。

## Solution
解决方案概述。

## User Stories
- As a <角色>, I want <功能>, so that <价值>

## Implementation Decisions
已决策的模块划分、接口、技术选型。

## Testing Decisions
哪些模块要测试、怎么测。

## Out of Scope
本次不涵盖的内容。
```

**输出**：完整的 PRD 文档。

---

### 阶段 5 — 拆解任务

将 PRD 拆解为可独立执行的垂直切片，按依赖顺序排列。

每个 issue 是一刀切穿所有层的薄切片（schema → 逻辑 → 测试），不是切一层。

两种类型：
- **AFK** — 可直接实现并合并
- **HITL** — 需要你做决策

**输出**：按依赖顺序排列的 issue 清单，标注类型和前置依赖。

---

## 什么时候用

- 开始一个中型以上功能前
- 方案不确定，需要系统性地审视
- 需要一份正式的 PRD 和任务拆解
- 团队协作时需要分派独立任务

## 案例

```
你：我想给系统加一个缓存层，帮我规划一下
Claude：好的，先从最基本的开始——你要缓存什么数据？
       我建议缓存数据库查询结果，而不是计算中间结果。
       你觉得呢？
你：对，查库结果
Claude：缓存粒度呢？
       A) 全表缓存
       B) 逐行缓存
       C) 按查询参数缓存
       我推荐 C，命中率最高。
你：选 C
...

[进入阶段 2 → 阶段 3 → ... → 阶段 5]

Claude：规划完成，共拆出 4 个 issue：
       1. [AFK] 缓存基础设施 — 基础抽象 + TTL 策略
       2. [AFK] 查询缓存接入 — 缓存装饰器实现
       3. [AFK] 失效策略 — 写入时缓存失效
       4. [HITL] 缓存监控 — 命中率指标 + 告警阈值
```

## 完成后 — 确认后推进

规划闭环（阶段 1-5 走完、PRD + 任务清单产出）后，先展示最终规划摘要。**共享理解确认只授权生成规划产物，不等于授权修改代码。**

只有用户再次明确确认 PRD 和任务清单可执行，且没有 HITL 阻塞，才按工作流路由进入下一步：

- **默认下一步**：`/4-开发` — 按任务清单顺序 TDD 实现。
- **例外分支**：阶段 1-3 仍有不确定的设计决策（数据模型/状态机/接口取舍拿不准）→ 先 `/3-原型` 验证，再 `/4-开发`。
- **暂停条件**：用户尚未授权执行、任务清单里有 HITL 项需拍板，或方案共识未达成——先问再进，不硬闯。

获得执行授权后，一句话报出：「→ 进入 /4-开发」，然后执行该 skill 的 SKILL.md。
