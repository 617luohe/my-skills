# 4-开发 — TDD 编码实现

红-绿-重构循环开发，pytest 驱动，先写失败测试再写实现。

## 编码准则（贯穿全程）

在编码过程中遵循以下四条原则：

### 1. 先想后写
编码之前说出你的假设、列出多种解读、如果有更简单的方案就说出来。

### 2. 简单第一
最少代码解决问题。没有要求的功能不加，只用一次的逻辑不抽象，没人要求的灵活性不做。

### 3. 手术刀式改动
只碰必须碰的，只清理自己制造的垃圾，不顺手改进旁边的代码。

### 4. 目标驱动执行
把模糊任务转成可验证的目标。多步骤时给出简短计划和验证项。

---

## MUST 规则（贯穿全程的四条准则）

1. **先想后写。** 编码前说出假设，不确定就问。
2. **简单第一。** 最少代码解决问题，不做投机功能。
3. **手术刀式改动。** 只碰必须碰的，不顺手改旁边的。
4. **目标驱动。** 模糊任务转可验证目标，多步骤先列计划。
5. **绝不在 RED 时重构。** 全部变绿后才检查提取重复/加深模块。
6. **测试通过公共接口验证行为，不验证实现细节。**
7. **遇 HITL 任务时暂停决策。** 如果上游（1-规划）标注任务为 `[HITL]`，执行到需要用户决策的点时：
   - 展示当前进度和待决策项（如"告警阈值设多少？推荐 95%"）
   - 等待用户输入后继续
   - 不替用户做 HITL 决策

## TDD 流程

### 核心理念

测试应该**通过公共接口验证行为**，而不是验证实现细节。代码可以完全重写，测试不应该变。
优先写集成风格测试：走真实调用路径，验证用户可感知行为。

### 反模式：水平切片

```
错误（水平）:
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正确（垂直）:
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
```

### 步骤

1. **规划** — 先读术语与决策上下文（`CONTEXT.md` / ADR），确认接口设计，列出要测试的行为（不是实现步骤），等你批准。
   
   **测试策略确认**（根据任务规模调整）：
   - **简单任务**（单文件小改动、<50行代码）：
     - 确认是否需要测试："这个改动需要写测试吗？推荐：[是/否]，理由：[...]"
     - 如果不需要：跳过测试，直接实现 + 手动验证
     - 如果需要：至少写 1 个关键路径测试（happy path）
   
   - **中大任务**（新功能、重构、>50行代码）：
     - **你不可能测试一切**——确认你最关心的行为和关键路径，聚焦在复杂逻辑和边界上
     - 列出 3-5 个核心行为，按优先级排序
     - 等你批准后再开始

2. **示踪弹** — 写一个测试确认一件事 → 最少代码让它通过，证明路径可行
3. **递增循环** — 对每个剩余行为：
   - **RED**: 写下个测试 → 确认失败
   - **GREEN**: 最少代码让它通过
   - 一次一个测试，不提前写下一个，不超前实现
4. **重构** — 全部变绿后检查：提取重复、加深模块、应用 SOLID
   - **绝不在 RED 时重构**

### 每个周期自检

- [ ] 测试描述行为，不是实现
- [ ] 只用公共接口
- [ ] 重构后测试不变
- [ ] 代码量刚好通过当前测试
- [ ] 没有投机功能

### 规划阶段补充清单

- [ ] 和你确认了公开接口长什么样
- [ ] 和你确认了优先测试哪些行为
- [ ] 识别了可提炼的深模块机会（小接口封装高复杂度）
- [ ] 测试点按行为清单组织，不按实现步骤组织

## 什么时候用

- 开始实现一个需要高可靠性的功能
- 修复 bug 时先写重现测试再修复
- 模块接口已确定，需要扎实实现

## 案例

```
你：用 TDD 实现一个计算器 add 方法
Claude：接口确认：def add(a: int, b: int) -> int

       RED: test_add_returns_sum → 失败（add 不存在）
       GREEN: def add(a, b): return a + b → 通过

       还要加用例吗？负数、零、大数？
你：加个负数测试
Claude: RED: test_add_with_negative → 已有实现，直接通过
        无需重构，代码已最简单。
```

## 完成后 — 自动推进

功能实现完、测试全绿后，按工作流路由自动往下走，不停下来等用户点名：

- **开发涉及可量化目标**（性能优化/可靠性提升/非确定性bug修复）→ 完成后转 `/7-调试` 构建回路验证指标达标，验证通过后再 → `/5-检查`
  - 判断信号：上游来自 7-调试 或 6-优化，且目标含可量化指标（延迟<Xms、复现率<Y%、内存<ZMB、覆盖率>N%）
  - 流程：4-开发实施 → 7-调试验证达标 → 5-检查审查质量
- **默认下一步 → `/5-检查`** — 双轴审查把关，这是中大功能环的固定下一环。
- 开发中撞到难复现的 bug / 性能问题 → 转 `/7-调试`。
- 全部完成、要保存进度 → `/8-版本管理`；会话收尾 → `/9-最后整理`。

推进前一句话报出下一步 skill，再执行。
