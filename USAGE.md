## 辩证矛盾分析法 — 矛盾分析思维框架

**场景**：遇到复杂问题需要系统性分析时使用。适合宏观判断、战略规划、根因分析、逆境突破、团队动员、决策复盘等场景。

**阶段**：独立方法论（无阶段编号）

**触发方式**：`/辩证矛盾分析法` 或 `/use-skills 矛盾分析`

**核心流程**（六步法）：
1. **调查研究** — 先了解实际情况再发言
2. **定性** — 区分战略/战术、全局/局部
3. **矛盾分析** — 找主要矛盾，分内因外因，一分为二
4. **阶段划分** — 相持→反攻→收官，不同阶段不同策略
5. **策略制定** — 集中力量、统一战线、试点推广
6. **持续复盘** — 定期反思、知错即改

**语言风格**：比喻式（人民的语言）为主，搭配设问式、对比式、断言式、鼓舞式等多种句式。困境场景优先用经历共情式。

**违禁词**：严禁第一人称口吻、严禁政治立场表态、严禁敏感话题。

**对话示例**：
```
你：/辩证矛盾分析法 我们团队在 2026-07 的交付效率下降，开发节奏越来越慢
Claude：[调查研究] 我先看看你们团队的组织结构和技术栈……
        [定性] 看起来是组织协作问题，不是技术问题。
        [矛盾分析] 主要矛盾在于：需求和交付能力之间的对抗性矛盾。
        内因是团队沟通效率低，外因是需求方不断加急单。
        [策略] 建议分三步走：先统一需求入口（统一战线）……
```

```
你：/use-skills 帮我用辩证分析法看看这个项目的架构问题
Claude：→ 分析需求：用矛盾分析法分析项目架构
       → 匹配技能：辩证矛盾分析法
       → 开始执行...
```

## 老油条 — 交付节奏管理

**场景**：面向甲方交付 Python 项目时，开发速度快于交付节奏，需要逐步展示优化成果、控制交付节奏、体现持续工作量。

**阶段**：独立方法论（无阶段编号）

**触发方式**：`/老油条 <你的需求>`（自然语言入口）

**核心机制**：
- **分支开关** — `branch_config.json` 中 `{"sort": "timsort"}` 式的键值对，控制每个模块使用哪个实现版本
- **注册表** — 代码中的 dict 映射（如 `SORT_ENGINES = {"basic": BasicSort, "timsort": TimSort}`），声明模块的所有可选实现
- **叙事线** — 升级项分类：性能、功能、体验、安全
- **交付路线图** — 面向甲方的多次交付计划
- **交付说明** — 每次交付自动生成的甲方可读文档

**五种模式**（由自然语言自动路由）：
1. **初始化** — 扫描项目注册表 → 创建 branch_config.json → 生成路线图
2. **分析建议（黄金路径）** — 扫描 → 计算增量 → 按叙事线给一条建议 → 确认 → 生成
3. **精确切换** — 手动指定模块切换到特定实现
4. **状态查看** — 只读展示当前各模块分支状态
5. **交付说明生成** — 生成甲方可读的交付文档

**核心设计原则**：一条建议，不列菜单。默认路径下直接给出最优建议，用户确认即可。

**对话示例**：
```
你：/老油条 快到交付节点了，这次交什么

Claude：[扫描 → 增量计算 → 叙事线分类...]

       老油条建议本次交付 — 性能优化线
       ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
       模块   当前 → 建议        收益摘要
       sort   basic → timsort    排序耗时降低 40-60%
       cache  memory → redis     首次加载减少 200ms
       ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
       涉及模块：2 个  叙事线：性能优化  风险等级：低
       ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
       是否按此方案交付？ [Y] 是  [N] 跳过  [M] 手动调整  [S] 换叙事线

你：Y

Claude：已交付 — 性能优化线
       ✓ sort:  basic → timsort
       ✓ cache: memory → redis
       交付说明已生成 → delivery-notes/2026-06-29-性能优化.md
```

```
你：/老油条 帮我初始化

Claude：[扫描项目 Python 文件...]
       发现 4 个注册表：
         sort     → 3 个实现 (basic, timsort, parallel)     [性能]
         cache    → 3 个实现 (memory, redis, memcached)     [性能]
         renderer → 2 个实现 (dom, webgl)                   [体验]
         auth     → 2 个实现 (basic, oauth)                 [安全]

       已创建 branch_config.json，全部使用基础实现。
       交付路线图已生成，共 4 次交付计划。
```

```
你：/老油条 把排序切成 parallel

Claude：验证通过，sort 有 parallel 实现。
       分支切换完成
       ✓ sort: timsort → parallel
```

**输出产物**：
| 文件 | 说明 |
|------|------|
| `branch_config.json` | 分支开关配置（核心文件） |
| `.laoyoutiao/delivery-log.json` | 交付历史追踪 |

## 自动迭代 — 多轮闭环编排

**场景**：有一个明确目标但需要多轮迭代才能闭环——比如开发→审查→发现问题→修复→再审查。不想手动记住每轮该调哪个 skill。

**位置**：编排层（与 use-skills 平级），调用 use-skills 做单轮路由。

**触发方式**：`/自动迭代 <目标描述>` 或自然语言「自动帮我<做某事>」

**可选参数**：`--manual`（每轮确认）、`--max-rounds=N`（熔断阈值，默认15）、`--skip-probe`（跳过探查）、`--checkpoint=<stage>`（阶段性暂停）

**对话示例**：
```
你：/自动迭代 给 user-service 加上 Redis 缓存，端到端完成

Claude：[阶段 1] 探查：Python + SQLAlchemy + Redis 已安装
       [阶段 2] 目标清晰，跳过追问
       [阶段 3] 拆出 4 个任务：缓存抽象层 → 查询缓存接入 → 失效策略 → 监控
       [阶段 4] 迭代执行：
         迭代 1：缓存抽象层 → ✅ pass
         迭代 2：查询缓存接入 → ❌ fail（线程不安全 + 缓存键冲突）
         决策：追加 2a（线程安全）+ 2b（缓存键修复）
         迭代 3-4：修复子任务 → ✅ pass
         迭代 5-7：剩余任务 → ✅ pass
       [阶段 5] 收尾：4 原始任务 + 2 追加，7 轮迭代，全部通过
       LOOP_LOG.md 已写入
```

```
你：/自动迭代 迁移认证到 OAuth --manual

Claude：[迭代 2 完成] 门禁 ✅ pass
       推荐下一步：任务 3（Token 刷新）
       继续？[Y / n / 改方向]
你：Y
```

**输出产物**：
| 文件 | 说明 |
|------|------|
| `LOOP_LOG.md` | 完整迭代记录（三层：摘要 + 决策追溯 + 门禁状态） |
| 任务链实际产出 | 各子 Agent 产出的代码/文档/报告 |

## 0--headroom-learn — 会话失败学习器

**场景**：对话中反复出现同类错误——路径找不到、包名写错、命令权限不对、同样的操作重试 3 次以上。分析失败模式，自动生成纠正规则写入 CLAUDE.local.md，防止同类错误重复发生。

**阶段**：阶段 0 溢出技能（部署名 `0--headroom-learn`）

**触发方式**：`/0--headroom-learn` 或自然语言「分析失败模式」「会话复盘」「为什么老出错」

**核心机制**：
- **模式识别** — 扫描会话历史，识别路径错误、缺失依赖、格式错误、顽固重试、RTK 循环等 7 类失败模式
- **根因分析** — LLM 分析每组失败的根本原因和预防方案
- **规则生成** — 生成简洁可执行的纠正规则
- **默认写入 CLAUDE.local.md** — 个人学习不污染团队共享配置，标记 `<!-- headroom-learn: <date> -->`

**对话示例**：
```
你：/0--headroom-learn

Claude：luohe，扫描当前项目 10 个会话...
       发现 4 个失败模式：

       ⚠️ RTK 循环: grep 为空后重复执行 — 浪费 12,500 tokens
       路径错误: 8次在 src/ 找文件，实际在 packages/
       包管理器: 5次用 npm，项目实际用 pnpm
       Python venv: 3次忘了激活虚拟环境

       ✅ 4 条规则已写入 CLAUDE.local.md
```

## 0--headroom-compress — 上下文压缩工具箱

**场景**：工具输出太大（搜索结果、日志、批量文件读取），或上下文窗口开始膨胀时，用 Headroom MCP 工具压缩内容并保留关键信息，节省 50-90% token。

**阶段**：阶段 0 溢出技能（部署名 `0--headroom-compress`）

**触发方式**：`/0--headroom-compress` 或自然语言「压缩一下」「token太多」「精简输出」

**三个工具**：
| 工具 | 用途 |
|------|------|
| `headroom_compress` | 压缩大段输出，保留函数签名/错误/关键字段 |
| `headroom_retrieve` | 按 hash 检索被压缩的原始内容 |
| `headroom_stats` | 查看会话中省了多少 token / 成本 |

**压缩策略**：
- 代码文件 → AST 感知（保留签名、imports，压缩实现细节）
- JSON/API → SmartCrusher（保留错误和关键字段，去重）
- 日志/文本 → Kompress ML + TextCrusher（保留错误行和堆栈）

**对话示例**：
```
你：搜索了整个项目的所有 Python 文件，输出太大

Claude：luohe，搜索结果太大，我先压缩一下。
       → headroom_compress → 关键 15 个文件，省了 73% token
       需要完整列表时用 headroom_retrieve(hash="abc123")
```

## 1-规划 — 方案设计与任务拆解

**场景**：写代码前方案不确定，需要系统性地理清需求、设计接口、输出 PRD、拆解任务。适合中型以上功能。

**触发方式**：`/1-规划`

**新增特性**（整合自 grill-with-docs）：
- 🔍 **文档发现** — 自动扫描 CONTEXT.md / ADR，交叉验证方案与已有领域模型
- 🎯 **场景压力测试** — 用具体场景探测概念边界和逻辑漏洞
- ✅ **代码交叉验证** — 读实际代码核实你的描述
- 📝 **内联 CONTEXT.md 更新** — 术语确认后即时更新文档
- 📋 **ADR 决策记录** — 对关键架构决策创建轻量记录

**包含 6 个阶段**（按顺序推进）：
1. 方案追问 — 先扫描 CONTEXT.md/ADR 了解领域现状，再用决策树穷举追问（含场景压力测试和代码交叉验证）
2. 领域术语 — 提取术语表，实时更新 CONTEXT.md，标记模糊词汇和歧义
3. 决策记录 — 对难逆转、少见、有真实取舍的决策创建轻量 ADR
4. 接口设计 — 并行生成多种方案对比
5. 输出 PRD — 产出结构化需求文档
6. 拆解任务 — 拆为可独立执行的垂直切片

**对话示例**：
```
你：/1-规划 帮我想想怎么做订单取消功能
Claude：[文档发现] 发现已有 CONTEXT.md，其中定义 Order 状态包括 PENDING、PAID、REFUNDING。
       你的 CONTEXT.md 定义"取消"是用户主动撤销未支付订单。
       但你说"取消"——是不是也包括已支付订单的取消？那是两个不同的流程。

你：两种都要支持

Claude：[追问] 已支付订单取消后，退款怎么处理？
       我推荐自动原路退款 + 异步确认。你觉得呢？
你：可以

Claude：[场景测试] 如果一个订单分多次发货，部分发货后取消——已发货部分是否要单独退款？
...

[进入阶段 2 — 术语确认后即时更新 CONTEXT.md]
[进入阶段 3 — 为"订单取消"和"部分退款"流程创建 ADR]
```
