## 0-辩证矛盾分析法 — 矛盾分析思维框架

**场景**：遇到复杂问题需要系统性分析时使用。适合宏观判断、战略规划、根因分析、逆境突破、团队动员、决策复盘等场景。

**阶段**：阶段 0 — 基础哲学方法论（思维工具）

**触发方式**：`/0-辩证矛盾分析法` 或 `/use-skills 矛盾分析`

**核心流程**（六步法）：
1. **调查研究** — 先了解实际情况再发言
2. **定性** — 区分战略/战术、全局/局部
3. **矛盾分析** — 找主要矛盾，分内因外因，一分为二
4. **阶段划分** — 相持→反攻→收官，不同阶段不同策略
5. **策略制定** — 集中力量、统一战线、试点推广
6. **持续复盘** — 定期反思、知错即改

**语言风格**：比喻式（人民的语言）为主，搭配设问式、对比式、断言式、鼓舞式等多种句式。困境场景优先用经历共情式。

**违禁词**：严禁第一人称口吻、严禁政治立场表态、严禁敏感话题。

**对话示例**：
```
你：/0-辩证矛盾分析法 我们团队最近效率低下，开发节奏越来越慢
Claude：[调查研究] 我先看看你们团队的组织结构和技术栈……
        [定性] 看起来是组织协作问题，不是技术问题。
        [矛盾分析] 主要矛盾在于：需求和交付能力之间的对抗性矛盾。
        内因是团队沟通效率低，外因是需求方不断加急单。
        [策略] 建议分三步走：先统一需求入口（统一战线）……
```

```
你：/use-skills 帮我用辩证分析法看看这个项目的架构问题
Claude：→ 分析需求：用矛盾分析法分析项目架构
       → 匹配技能：0-辩证矛盾分析法
       → 开始执行...
```

## 0-初始化CLAUDE — CLAUDE.md 初始化器

**场景**：开始新项目时初始化 AI 协作规则，或重置现有项目的 CLAUDE.md。一键生成，无需交互。

**触发方式**：`/0-初始化CLAUDE`

**生成内容**：

**对话示例**：
```
你：帮我初始化这个项目的 CLAUDE 文件

       ✅ CLAUDE.md 已生成，包含称呼规则 + Karpathy 准则。
```

## 0-构建索引 — 构建项目知识图谱

**场景**：将代码、文档、PDF、图片、视频等构建为可查询的知识图谱。适合接手大型陌生项目、团队共享架构地图、或对已有 `graphify-out/` 做自然语言查询。

**阶段**：阶段 0 — 环境准备 / 项目理解（索引构建）

**触发方式**：`/0-构建索引` 或 `/use-skills 构建索引`

**前置依赖**：
```powershell
uv tool install graphifyy
# 可选 extras: uv tool install "graphifyy[pdf,office,video,mcp]"
```

**输出**（`graphify-out/`）：
- `graph.html` — 浏览器交互式图谱
- `GRAPH_REPORT.md` — God Nodes、意外连接、建议问题
- `graph.json` — 完整图数据，供 `graphify query` 使用

**对话示例**：
```
你：/0-构建索引 .
Claude：Corpus: 128 files · ~45,000 words
       [AST + 语义提取 → 聚类 → 生成报告]
       God Nodes: AuthService, DatabasePool, ...
       最有趣的问题：认证失败时错误如何传播到 API 响应？要追踪吗？

你：/0-构建索引 query "订单状态机有哪些状态转换？"
Claude：→ graphify query "订单状态机有哪些状态转换？"
       [基于 graph.json 回答，引用 source_location]
```

## 1-规划 — 方案设计与任务拆解

**场景**：写代码前方案不确定，需要系统性地理清需求、设计接口、输出 PRD、拆解任务。适合中型以上功能。

**触发方式**：`/1-规划`

**新增特性**（整合自 grill-with-docs）：
- 🔍 **文档发现** — 自动扫描 CONTEXT.md / ADR，交叉验证方案与已有领域模型
- 🎯 **场景压力测试** — 用具体场景探测概念边界和逻辑漏洞
- ✅ **代码交叉验证** — 读实际代码核实你的描述
- 📝 **内联 CONTEXT.md 更新** — 术语确认后即时更新文档
- 📋 **ADR 决策记录** — 对关键架构决策创建轻量记录

**包含 6 个阶段**（按顺序推进）：
1. 方案追问 — 先扫描 CONTEXT.md/ADR 了解领域现状，再用决策树穷举追问（含场景压力测试和代码交叉验证）
2. 领域术语 — 提取术语表，实时更新 CONTEXT.md，标记模糊词汇和歧义
3. 决策记录 — 对难逆转、少见、有真实取舍的决策创建轻量 ADR
4. 接口设计 — 并行生成多种方案对比
5. 输出 PRD — 产出结构化需求文档
6. 拆解任务 — 拆为可独立执行的垂直切片

**对话示例**：
```
你：/1-规划 帮我想想怎么做订单取消功能
Claude：[文档发现] 发现已有 CONTEXT.md，其中定义 Order 状态包括 PENDING、PAID、REFUNDING。
       你的 CONTEXT.md 定义"取消"是用户主动撤销未支付订单。
       但你说"取消"——是不是也包括已支付订单的取消？那是两个不同的流程。

你：两种都要支持

Claude：[追问] 已支付订单取消后，退款怎么处理？
       我推荐自动原路退款 + 异步确认。你觉得呢？
你：可以

Claude：[场景测试] 如果一个订单分多次发货，部分发货后取消——已发货部分是否要单独退款？
...

[进入阶段 2 — 术语确认后即时更新 CONTEXT.md]
[进入阶段 3 — 为"订单取消"和"部分退款"流程创建 ADR]
```

## 3-原型 — 快速原型验证

**场景**：设计方案中有不确定的决策（数据模型、状态机、算法、API 设计），先用一次性代码快速验证，再进入正式开发。

**触发方式**：`/3-原型`

**两种模式**：
- **逻辑原型** — CLI 脚本验证数据模型/状态机/算法
- **UI 原型** — Web 页面验证交互（按需使用）

**对话示例**：
```
你：/3-原型 帮我验证一下这个订单状态机设计
Claude：你的设计中，订单状态是 pending → paid → refunding → refunded，
       退款完成后还能再次退款吗？
       我先写个原型跑一下看看...
       [创建 prototype_order_state.py]
       State: pending → pay() → State: paid
       State: paid → refund() → State: refunding
       State: refunding → complete_refund() → State: refunded
       State: refunded → refund() → Error: 已退款订单不可再退款
       验证通过，可以进入 4-开发 了。
```

## 4-开发 — TDD 编码实现

**场景**：实现功能或修复 bug 时使用。pytest 驱动，红-绿-重构循环，贯穿四条编程准则。

**触发方式**：`/4-开发`

**准则**：先想后写 → 简单第一 → 手术刀式改动 → 目标驱动

**流程**：规划 → 示踪弹 → RED→GREEN 递增循环 → 重构

**对话示例**：
```
你：用 TDD 实现一个计算器 add 方法
Claude：接口确认：def add(a: int, b: int) -> int
       RED: test_add_returns_sum → 失败
       GREEN: def add(a, b): return a + b → 通过
```

## 5-检查 — 代码审查与验收

**场景**：合并代码前审查质量，或者发现 bug 时报 issue。

**触发方式**：`/5-检查`

**两个模式**：
- **代码审查** — 双轴并行（Python 规范 + 需求符合度）
- **Bug 报告** — 你口述问题，Claude 探索代码并提交 GitHub issue

**对话示例**：
```

Claude：Standards 审查：

       Spec 审查：
       ❌ "退款原路返回" 未实现
```

## 6-优化 — 重构与架构改进

**场景**：代码越来越乱、模块太臃肿、测试难写时，系统性发现架构问题或制定重构计划。

**触发方式**：`/6-优化`

**两个模式**：
- **架构分析** — 发现浅模块，提出深化方案
- **重构计划** — 输出渐进式极小提交方案

**对话示例**：
```
你：payment_service.py 800 行了，帮我优化
Claude：支付、退款、对账、通知四个功能耦合在一起。
       建议分 5 步提取...
```

## 7-调试 — 结构化调试

**场景**：遇到难复现的 bug 或性能突然下降时，用六阶段流程系统性地排查根因。

**触发方式**：`/7-调试`

**六阶段**：构建反馈回路 → 复现 → 假设（3-5 条） → 工具验证 → 修复 + 回归测试 → 清理

**Python 工具**：breakpoint()、cProfile、py-spy、tracemalloc

**对话示例**：
```
你：订单支付接口偶尔返回 500，帮我调试
Claude：先构建回路——curl 循环调用 100 次...
       发现约 5% 失败。3 条假设：
       1. 连接池耗尽 2. 网关超时 3. 缓存穿透
```

## 9-最后整理 — 收尾与交接

**场景**：完成一个阶段、结束一天工作、或把项目交给别人时使用。做三件事：交接、知识整理、安全确认。

**触发方式**：`/9-最后整理`

**三步骤**：
1. **会话交接** — 创建交接文档，引用已有产物，不重复内容
2. **知识整理** — 尺寸检查 → 同步文档和记忆 → 自检清单
3. **安全护栏** — 确认 PreToolUse 钩子生效

**对话示例**：
```
你：今天做完了，帮我整理一下
Claude：交接文档 → handoff-a7k3.md
       知识同步 → CLAUDE.md + 路由表
       安全护栏 → 已配置
```
