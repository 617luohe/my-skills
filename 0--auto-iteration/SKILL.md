# 自动迭代 — 多轮闭环编排

输入一个目标，自动完成五阶段 PDCA 闭环推进。全自动模式默认；手动模式可选。

位于 skills 体系编排层，与 use-skills 平级。调用 use-skills 做单轮任务路由，不重复实现调度逻辑。

## MUST 规则

1. **五阶段按顺序推进，不跳跃。** 每阶段完成后写入日志。
2. **主 Agent 上下文永远精简。** 只持有日志引用 + 任务链 + 门禁状态表。每轮 spawn 独立子 Agent。
3. **门禁是唯一推进判据。** pass → 推进；fail → 决策引擎判断路径。不凭感觉推进。
4. **日志实时写入。** 每轮迭代结束立即更新 LOOP_LOG.md。不批量等最后。
5. **默认全自动推进。** 除非用户启动时声明"手动"/"需要确认"，否则不暂停询问。

## 阶段速查

| 阶段 | 名称 | 核心动作 | 输出 | 自动化 |
|------|------|----------|------|--------|
| 1 | 状态探查 | 扫描项目代码/文档，建立基线 | 基线 → 写入日志 | 全自动 |
| 2 | 目标澄清 | 模糊目标 → grill-me 追问；清晰目标 → 跳过 | 明确目标 | 自动判断清晰度 |
| 3 | 任务规划 | 基于目标+基线拆解任务链 | 任务链 → 写入日志 | 全自动 |
| 4 | 迭代执行 | PDCA 循环：取任务→spawn子Agent→门禁→决策推进 | 门禁报告 × N | 默认全自动，可选手动 |
| 5 | 收尾 | 最终日志落地 + 任务完成汇总 | LOOP_LOG.md + 汇总 | 全自动 |

## 触发方式

### 层 0 — 零参数（覆盖 90% 场景）

```
/自动迭代 <目标描述>
```

等价自然语言触发：
- `自动帮我<做某事>`
- `帮我自动完成<目标>`
- `持续改进<目标>`
- `迭代执行<目标>`

### 层 1 — 关键词推断

在目标描述中嵌入关键词，自动推断模式：

| 关键词 | 推断行为 |
|--------|----------|
| "手动"、"先问我"、"让我确认" | 切换为手动模式（每轮暂停确认） |
| "只规划"、"调研"、"只分析不写代码" | max-rounds=0，跳过迭代执行 |
| "不设上限"、"直到完美" | 熔断提到 25 轮 |
| "跳过探查" | 跳过状态探查阶段 |

### 层 2 — 显式 flag

```
/自动迭代 <目标描述> [--flag ...]
```

| Flag | 类型 | 默认 | 说明 |
|------|------|------|------|
| `--manual` | 布尔 | false | 每轮门禁判定后暂停，等待用户确认 |
| `--max-rounds=N` | 整数 | 15 | 熔断阈值，达此轮数强制终止 |
| `--skip-probe` | 布尔 | false | 跳过状态探查，假设主 Agent 已了解项目 |
| `--task-chain=<file>` | 路径 | 无 | 用户提供的任务链文件（JSON/MD），跳过任务规划 |
| `--checkpoint=<stage>` | 枚举 | 无 | 在指定阶段后暂停：`stage2`/`stage3`/`stage4` |
| `--context=<glob>` | glob | 自动 | 额外注入给子 Agent 的上下文文件范围 |

`--checkpoint` vs `--manual` 的区别：
- `--checkpoint=stage3`：只在任务规划后暂停一次，其余全自动
- `--manual`：每轮迭代后都暂停
- 可以组合：`--manual --checkpoint=stage2` = 目标澄清后停一次 + 后续每轮都停

## 日志系统

所有迭代过程写入单个文件 `LOOP_LOG.md`，位于项目根目录。

### 日志三层结构

```markdown
# LOOP_LOG — <目标摘要>

## 迭代摘要

| # | 阶段 | 任务 | 子 Agent | 门禁 | 决策 | 耗时 |
|---|------|------|----------|------|------|------|
| 1 | 4 | 缓存抽象层 | 4-开发 | ✅ pass | 推进 | 45s |
| 2 | 4 | 查询缓存装饰器 | 4-开发 | ❌ fail | 修复 | 62s |
| 3 | 4 | 修复：线程安全 | 4-开发 | ✅ pass | 推进 | 30s |

## 决策追溯

### 迭代 2 — 查询缓存装饰器 fail

**门禁报告**：
- 阻断项：`cache.py:45` 线程不安全（无锁共享 dict）
- 阻断项：`cache.py:78` 缓存键冲突（不同用户共享键空间）

**决策依据**：
- 问题属于实现层面，不涉及架构变更 → 追加修复子任务
- 引用：LOOP_LOG.md 迭代 1（缓存抽象层已通过，基础设施正确）
- 下一步：追加 2a（线程安全）+ 2b（缓存键修复），然后重新检查

## 门禁状态

当前任务链：
- [x] 1. 缓存抽象层 — ✅ passed（迭代 1）
- [x] 2. 查询缓存装饰器 — ❌ failed → 追加修复子任务
- [ ] 2a. 修复：线程安全 — 🔄 in_progress
- [ ] 2b. 修复：缓存键策略 — ⏳ pending
- [ ] 3. 失效策略 — ⏳ pending
- [ ] 4. 缓存监控 — ⏳ pending

总迭代轮数：2 | 熔断阈值：15 | 阻塞项：0
```

### 日志写入时机

- **阶段 1 结束**：写入基线
- **阶段 2 结束**：写入确认的目标
- **阶段 3 结束**：写入任务链
- **阶段 4 每轮结束**：追加迭代摘要行 + 决策追溯 + 更新门禁状态
- **阶段 5 结束**：写入最终汇总
- **任务链动态变化时**：实时更新门禁状态中的任务链

### 日志用途

- **人可读**：通过迭代摘要快速了解路径，决策追溯理解 AI 判断
- **Loop 自引用**：主 Agent 每轮读取日志中的门禁状态 + 决策追溯，调整优化方向
- **自动恢复**：启动时检测 LOOP_LOG.md 存在且门禁状态有未完成任务 → 自动从中断点恢复

## 门禁系统

### 门禁报告 Schema

子 Agent 每次执行完成必须产出结构化门禁报告：

```json
{
  "gate": "pass" | "fail",
  "issues": [
    {
      "severity": "blocker" | "warning",
      "file": "src/cache.py",
      "line": 45,
      "desc": "线程不安全：多线程共享 dict 无锁保护"
    }
  ],
  "summary": "2 个 blocker，1 个 warning",
  "artifacts": ["src/cache.py", "tests/test_cache.py"]
}
```

### 门禁判定规则

| 条件 | 门禁结果 | 动作 |
|------|----------|------|
| 无 issues 或仅有 warning | `pass` | 任务标记 passed，推进下一个任务 |
| 存在 blocker | `fail` | 决策引擎判断路径（见阶段 4） |
| 同一任务连续 fail ≥ 3 次 | `fail` | 标记任务 blocked，追加人工介入任务 |
| 总迭代轮数达 max-rounds | 熔断 | 强制终止，标记所有未完成任务 blocked，进入收尾 |

### 门禁状态机

```
pending → in_progress → passed  ──→ 任务完成
                     → failed  ──→ 决策: 修复/追加子任务/标记 blocked
                     → blocked ──→ 等待人工决策后重新入队
```

- `pending`：尚未开始
- `in_progress`：当前轮正在执行
- `passed`：门禁通过，任务完成
- `failed`：门禁不通过，待决策引擎处理
- `blocked`：需人工决策，暂停推进

## 阶段 1 — 状态探查

扫描项目现状，建立基线。全自动，用户无感知。

### 执行步骤

1. 读取项目根 `CLAUDE.md`（如存在），了解项目规则
2. 扫描目录结构（顶层 + 关键子目录）
3. 检查关键配置文件：`package.json` / `pyproject.toml` / `Cargo.toml` / `go.mod` 等
4. 检查依赖清单
5. 读取 `CONTEXT.md`（如存在），了解术语体系
6. 将以上内容写入 LOOP_LOG.md 的基线节

### 基线格式

```markdown
## 基线

- **项目类型**：Python（pyproject.toml）
- **关键依赖**：FastAPI, SQLAlchemy, Redis
- **目录结构**：src/（核心代码）, tests/（测试）, docs/（文档）
- **现有文档**：CLAUDE.md, CONTEXT.md（老油条项目）
- **探查时间**：2026-06-29 14:30
```

### `--skip-probe` 处理

如果用户声明 `--skip-probe`，跳过此阶段，日志中标注"基线：用户跳过探查"。

---

## 阶段 2 — 目标澄清

判断用户目标是否清晰。清晰则跳过追问，直接进入任务规划。

### 清晰度判断

**清晰目标信号**（跳过追问，直接进入阶段 3）：
- ✅ 含具体技术名词（"Redis"、"JWT"、"mypy"、"FastAPI"）
- ✅ 含明确动作范围（"所有查询结果"、"整个 auth 模块"、"全部类型注解"）
- ✅ 含可验证的完成标准（"不再有类型报错"、"延迟低于 50ms"）

**模糊目标信号**（触发 grill-me 追问）：
- ❌ 抽象形容词为主（"优化性能"、"改进代码质量"、"让代码更好"）
- ❌ 无具体范围（"重构一下"、"改一改"、"看一下"）
- ❌ 多目标可能冲突（"要快又要省内存又要好维护又要少依赖"）

### 模糊目标追问

当目标被判定为模糊时，启动 grill-me 决策树追问（引用 1-规划 阶段 1b 的追问模式）：

1. 一次只问一个问题，先给出推荐答案
2. 沿着决策树逐个决议，不跳跃
3. 场景压力测试：用具体场景探测边界
4. 退出条件：所有决策分支已决议，目标可操作化

### 目标确认后

将明确的目标写入 LOOP_LOG.md：

```markdown
## 目标

> 给 user-service 的数据库查询层添加 Redis 缓存。缓存范围为读多写少的查询接口，
> 缓存策略为按查询参数缓存，TTL 默认 5 分钟。不缓存写操作。
```

## 阶段 3 — 任务规划

基于明确目标 + 状态基线拆解任务链。

### 自动拆解

用户未提供任务链时，主 Agent 自动拆解：

1. 读入：目标 + 基线（项目类型、目录结构、现有文档）
2. 将目标分解为 3-8 个垂直切片，每个切片是穿通一层的完整任务（schema → 逻辑 → 测试）
3. 标注每个任务的类型（AFK/HITL）和前置依赖
4. 为每个任务设定门禁关注点（如"必须通过 CI"、"线程安全"等）
5. 写入日志的门禁状态节

### 用户路线评估

用户通过 `--task-chain=<file>` 提供任务链时：

1. 读取用户任务链文件
2. 评估：任务能否解决问题？依赖顺序是否合理？粒度是否可执行？
3. 如果合理 → 直接采用，注明"采用用户路线"
4. 如果存在问题 → 指出具体问题，给出修正建议，让用户选择：采用修正版 / 按自动拆解重来

### 任务链格式

```markdown
## 任务链

- [ ] 1. [AFK] 缓存基础设施 — Redis 连接 + 序列化抽象 + TTL 策略 — 前置：无
- [ ] 2. [AFK] 查询缓存装饰器 — 装饰器实现 + 缓存键生成 + 接入现有查询层 — 前置：1
- [ ] 3. [AFK] 写入失效策略 — 写操作触发缓存清除 — 前置：2
- [ ] 4. [AFK] 缓存监控埋点 — 命中率 + 延迟指标 — 前置：3
```

### `--checkpoint=stage3` 处理

如果设置了 `--checkpoint=stage3`，在此阶段完成后暂停，展示任务链让用户审核，确认后继续。

## 阶段 4 — 迭代执行（PDCA 循环体）

核心循环。按 PDCA 子循环推进任务链直到闭环或跳出。

### 循环入口

阶段 3 完成后，主 Agent 进入以下循环：

```
while 任务链有 pending 或 failed 任务:
    if 总迭代轮数 >= max_rounds:
        触发熔断 → 进入阶段 5
    if 所有任务 passed 或 blocked:
        break → 进入阶段 5
    执行一次 PDCA 迭代
```

### PDCA 子循环

每轮迭代走完整 PDCA：

#### P — Plan（选定任务）

1. 扫描任务链，找第一个 `pending` 或 `failed` 任务
2. 如果有 `failed` 任务：先决策修复路径（见下方决策引擎），生成修复子任务或回退原任务
3. 如果有 `blocked` 任务：跳过，取下一个可用任务
4. 如果全部 `blocked`：暂停，输出阻塞汇总，请求人工介入
5. 将选定任务标记为 `in_progress`

#### D — Do（执行任务）

**主控/工人 spawn 协议**：

主 Agent 通过 Skill 工具调用目标 skill，spawn 独立子 Agent。子 Agent 仅接收以下三样输入：

1. **本轮任务描述** — 从任务链中取，含任务名、范围、预期产出
2. **上一轮门禁报告** — 如果是修复任务，附上原始 fail 的任务的门禁报告
3. **相关文件引用** — 从任务描述中提取路径模式（如 `src/auth/*`），作为上下文注入

严禁传给子 Agent：
- ❌ 完整对话历史
- ❌ 完整 LOOP_LOG.md
- ❌ 其他任务的内容
- ❌ 超过 3 项背景材料

子 Agent 执行结束后，必须产出门禁报告（JSON schema 见门禁系统节）。

**任务路由规则**：

主 Agent 根据任务类型匹配目标 skill——**优先查 `use-skills` 的技能路由表**（唯一路由源），不在本 skill 内维护副本。按任务描述提取意图（开发/修复/审查/重构/分析/原型/文档…），对照该表取匹配 skill。改路由只改 use-skills。

**降级兜底**（部分安装场景）：如果 use-skills 未安装/不可读，回退到内置意图判断，不因缺表而瘫痪：

| 意图 | 回退 skill |
|------|-----------|
| 开发/实现 | 4-开发 |
| 修复/调试 | 7-调试 |
| 代码审查 | 5-检查 |
| 重构/优化 | 6-优化 |
| 分析/调研 | 2-分析 |
| 原型验证 | 3-原型 |

这张表是 use-skills 全量映射的最小工作子集，仅供 use-skills 缺失时兜底；正常情况一律以 use-skills 为准。同理，若回退目标 skill 也未安装，跳过该任务并在门禁报告标注"目标 skill 缺失"，继续推进其余任务。

#### C — Check（门禁检查）

1. 解析子 Agent 的门禁报告
2. 根据门禁判定规则（见门禁系统节）判定 pass/fail
3. 判定结果写入 LOOP_LOG.md 的迭代摘要和门禁状态

**门禁关注点分类**（阶段 3 为每个任务预设）：

**1. 代码质量门禁**（默认，始终检查）：
- 测试通过（子 Agent 执行 pytest，报告测试数/通过数/失败数）
- 类型安全（mypy/pyright 无新增错误）
- Lint 通过（ruff/eslint 无新增警告）
- 代码审查（5-检查 产出审查结论）

**2. 功能目标门禁**（按目标定义，阶段 2 澄清时设定）：
- **性能指标**：如目标"API 响应<100ms" → 门禁检查实测延迟
- **可靠性指标**：如"复现率<1%" → 门禁运行压力测试
- **资源消耗**：如"内存<500MB" → 门禁检查 profiler 输出
- **覆盖率**：如"测试覆盖率>80%" → 门禁检查 coverage 报告

**门禁设定规则**：
- 阶段 2（目标澄清）时，如果目标含可量化指标（如"延迟<100ms"、"复现率<1%"），在阶段 3（任务规划）为对应任务添加功能目标门禁
- 示例：目标"优化数据库查询到 50ms" → 任务"实现查询缓存" → 门禁关注点增加"查询延迟<50ms"

| 关注点示例 | 检查方式 |
|--------|----------|
| 测试通过 | 子 Agent 执行 pytest，报告测试数/通过数/失败数 |
| 类型安全 | mypy/pyright 无新增错误 |
| Lint 通过 | ruff/eslint 无新增警告 |
| 代码审查 | 5-检查 产出审查结论 |
| 性能达标 | benchmark 实测值 vs 目标值 |
| 覆盖率达标 | coverage report 实际值 vs 目标值 |

#### A — Act（决策推进）

门禁 pass 或 fail 后，决策引擎判断推进路径：

### 决策引擎

```
门禁 pass
├── 无 warning → 标记 passed，推进下一个任务
└── 有 warning → 标记 passed（附注），推进下一个任务

门禁 fail
├── 阻断项 ≤ 2 个，首次 fail → 任务回退 in_progress，追加 1 个修复子任务
├── 阻断项 ≤ 2 个，第 2 次 fail → 尝试不同修复策略，追加 1 个备选修复子任务
├── 阻断项 ≤ 2 个，第 3 次 fail → 标记 blocked，追加人工介入任务
├── 阻断项 > 2 个 → 标记 blocked，追加人工介入任务（问题太多，AI 可能方向错了）
└── 门禁报告含"需人工决策"标记 → 标记 blocked，暂停

任务 blocked
├── 用户响应后 → 重新入队为 in_progress
└── 用户无响应 → 跳过此任务，推进下一个可用任务
```

### 终止条件判断

每轮结束后检查：

1. **质量通过**：任务链全部 `passed` → 进入阶段 5（正常出口）
2. **人工接管**：用户在任何阶段说"停"/"够了"/"我来" → 立即进入阶段 5
3. **上限熔断**：总迭代轮数达 `max-rounds`（默认 15） → 标记剩余任务 blocked，进入阶段 5
4. **全部阻塞**：所有未完成任务均为 `blocked` → 输出阻塞汇总，进入阶段 5

### 日志更新

每轮迭代结束即时写入 LOOP_LOG.md：

1. **迭代摘要**追加一行
2. **决策追溯**追加本轮决策依据和门禁报告摘要
3. **门禁状态**更新任务链状态

### `--checkpoint=stage4` 处理

设置后每轮迭代暂停，等待用户确认。等效于 `--manual`。区别是 checkpoint 只在阶段 4 暂停，而 manual 贯穿全流程。

## 手动模式

通过 `--manual` flag 或自然语言关键词（"手动"、"先问我"、"让我确认"）触发。

### 手动模式行为

默认全自动模式下，每轮门禁 pass/fail 后决策引擎自动判断推进方向。手动模式下：

**每轮迭代的暂停点**：

```
📋 第 3 轮迭代完成

门禁报告：❌ fail
- 阻断项：cache.py:45 线程不安全
- 阻断项：cache.py:78 缓存键冲突

推荐下一步：追加 2 个修复子任务（2a 线程安全 + 2b 缓存键），然后重新检查任务 2

任务链状态：
  1. ✅ 缓存抽象层
  2. ❌ 查询缓存装饰器（fail — 待修复）
  3. ⏳ 失效策略
  4. ⏳ 缓存监控

继续？[Y / n / 改方向]
```

- 用户回复 `Y` / `继续` → 按推荐路径执行
- 用户回复 `n` / `跳过` → 跳过当前任务，推进下一个
- 用户回复 `改方向：<说明>` → 按用户指定路径执行
- 用户回复 `停` / `够了` → 终止循环，进入收尾

### 手动模式 vs `--checkpoint` 的关系

两者可独立使用：

| 组合 | 行为 |
|------|------|
| `--manual` | 每轮迭代后都暂停确认 |
| `--checkpoint=stage3` | 仅任务规划后暂停一次 |
| `--manual --checkpoint=stage2` | 目标澄清后暂停一次 + 后续每轮暂停 |
| 无 flag | 全自动到底 |

## 边缘情况处理

### 空任务链
探查后发现目标无需执行（如"代码已满足目标"）→ 直接进入收尾，汇总标注"无需执行"。

### 子 Agent 执行异常
- Spawn 失败 / 超时 / 无响应 → 重试 1 次
- 重试仍失败 → 标记当前任务 blocked，原因："子 Agent 执行异常"
- 继续推进下一个可用任务

### 自动恢复
启动时检测到 `LOOP_LOG.md` 存在且门禁状态有未完成任务：
1. 读取日志中的基线 + 目标 + 任务链
2. 跳过阶段 1-3，直接进入阶段 4
3. 从第一个非 `passed` 且非 `blocked` 的任务开始
4. 日志中标注"[自动恢复]"

### 熔断触发
- 输出："已达最大迭代轮数（<max-rounds>）。以下任务未完成：..."
- 保留日志和已完成代码，用户可手动处理剩余任务或提高 max-rounds 重跑

### 目标漂移
子 Agent 产出的门禁报告与原始目标不匹配时（如"加了缓存但没接入查询层"）：
- 标记为门禁 fail，阻断项："产出偏离目标——缺少查询层接入"
- 追加修复子任务补充遗漏

### 任务链空洞
决策引擎追加子任务后，检查是否有未被覆盖的原任务依赖 → 如有，自动补全依赖链

## 自然语言推断

在目标描述中自动识别以下信号，推断模式：

### 模式推断

| 关键词 | 推断 | 优先级 |
|--------|------|--------|
| "手动"、"先问我"、"让我确认"、"每个阶段确认" | `--manual` | 高 |
| "只规划"、"调研"、"只分析"、"不写代码"、"只看不改" | max-rounds=0 | 高 |
| "不设上限"、"直到完美"、"做到满意为止" | max-rounds=25 | 中 |
| "快速迭代"、"边做边改"、"快点" | gate 宽松（warning 不阻止） | 低 |
| "仔细审"、"严格检查"、"这个很重要" | gate 严格（每个 warning 都审查） | 低 |
| "并行搞"、"同时做" | 尽量并行 spawn | 中 |

### 冲突处理

当关键词推断与显式 flag 冲突时，**显式 flag 优先**。例如：
- "不设上限搞好" + `--max-rounds=5` → 采用 5（显式 flag 覆盖推断）

### 推断日志

所有自然语言推断写入 LOOP_LOG.md：

```markdown
## 推断

- 检测到"手动" → 切换为手动模式
- 未检测到其他信号 → 其余采用默认值
```

## 什么时候用

- 一个明确目标需要多轮迭代才能闭环
- 不想手动记住任务进度和门禁状态
- 需要可追溯的迭代记录（LOOP_LOG.md）
- 希望 AI 自动调度多个 skill 完成一个完整目标

## 什么时候不用

- 单次任务 → 用 `/use-skills` 直接路由
- 完全不确定要做什么 → 先用 `/1-规划` 理清需求
- 只需要代码审查 → 直接用 `/5-检查`
