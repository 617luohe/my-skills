# 自动迭代 — 完整术语表

## 规范术语

| # | 术语 | 英文 | 定义 | 别用 |
|---|------|------|------|------|
| 1 | 自动迭代 | Auto-Iterate | 多轮编排 skill，接管任务链按 PDCA 自动推进直到闭环或跳出 | loop、循环任务、自动化工作流 |
| 2 | 任务链 | Task Chain | 有依赖顺序的可执行任务列表，每项是穿通一层的垂直切片。非静态，可动态追加 | pipeline、检查清单、todo list |
| 3 | 任务项 | Task Item | 任务链中的单个任务，对应一个独立子 Agent 执行 | step、阶段 |
| 4 | 任务状态 | Task Status | 每个任务项的当前状态：`pending \| in_progress \| passed \| failed \| blocked` | 进度、完成度 |
| 5 | 门禁 | Gate | 任务完成后的二元判定节点（pass/fail），决定推进方向 | checkpoint、review、审批 |
| 6 | 门禁报告 | Gate Report | 子 Agent 完成后的结构化产出：`{gate, issues[{severity, file, desc}], summary}` | 输出、结果、结论 |
| 7 | 阻断项 | Blocker | severity=blocker 的问题，阻止门禁通过 | 严重 bug、致命错误 |
| 8 | 警告项 | Warning | severity=warning 的问题，不阻止门禁通过但需关注 | 小问题、建议 |
| 9 | 熔断 | Circuit Break | 达到最大迭代轮数后的强制终止。针对总迭代轮数，非单任务重试 | limit、超时、强制退出 |
| 10 | 迭代 | Iteration | PDCA 循环中的一轮：取任务→执行→门禁→决策推进 | 循环轮次、round、cycle |
| 11 | 主 Agent | Master Agent | 持有日志+任务链+门禁状态，负责决策引擎和下一轮调度。始终存活，上下文精简 | 编排器、控制器、loop master |
| 12 | 子 Agent | Sub Agent | 每次迭代中 spawn 执行具体任务的独立 Agent（上下文隔离） | worker、执行者、任务代理 |
| 13 | 状态探查 | State Probe | 五阶段之第 1 阶段：读取项目文档和代码形成基线 | 代码扫描、前置探查、context gathering |
| 14 | 目标澄清 | Goal Clarification | 五阶段之第 2 阶段：grill-me 决策树追问至目标明确 | 需求确认、需求澄清 |
| 15 | 任务规划 | Task Planning | 五阶段之第 3 阶段：基于目标+基线拆解任务链 | 任务拆解、work breakdown |
| 16 | 迭代执行 | Iteration Loop | 五阶段之第 4 阶段：PDCA 循环体 | 主循环、执行阶段 |
| 17 | 收尾 | Wrap Up | 五阶段之第 5 阶段：输出最终日志+任务完成汇总 | 结束、收工 |
| 18 | 日志 | Log | 单个 Markdown 文件（`LOOP_LOG.md`），三层：摘要+追溯+门禁 | 记录、log、运行记录 |
| 19 | 自动模式 | Auto Mode | 默认模式，决策引擎全自动推进 | 全自动、无人值守 |
| 20 | 手动模式 | Manual Mode | 可选模式，每轮决策后需用户确认 | 审查模式、交互模式 |
| 21 | 阻塞 | Blocked | 任务状态。需人工决策才能继续，不退出循环 | 卡住、挂起 |
| 22 | 目标 | Goal | 用户提供的自然语言任务最终描述。唯一必需输入 | 需求、目的 |
| 23 | 基线 | Baseline | 状态探查产出的项目当前状态快照 | 现状、初始状态 |
| 24 | 决策引擎 | Decision Engine | 主 Agent 的核心逻辑：基于日志+门禁报告决定下一轮路径 | 路由器、调度器 |

## 关系图

```
自动迭代
├── 状态探查 ──→ 基线 → 写入日志
├── 目标澄清 ──→ 目标（grill-me 追问）
├── 任务规划 ──→ 任务链 [任务项, 任务项, ...]
├── 迭代执行 ──→ 迭代 × N
│   ├── 主Agent (决策引擎)
│   │   ├── 读取: 日志 + 任务链 + 门禁报告
│   │   ├── 决策: 推进/修复/追加/上报
│   │   └── 启动: 子Agent
│   ├── 子Agent → 门禁报告
│   ├── 门禁: pass → passed / fail → 决策
│   ├── 阻塞: 需人工 → blocked → 暂停
│   └── 熔断: 达上限 → 强制退出
├── 自动模式: 全自动推进（默认）
├── 手动模式: 每轮确认（声明启用）
└── 收尾 ──→ 最终日志 + 完成汇总
```

## 状态机：任务项生命周期

```
pending → in_progress → passed  ──→ 任务完成
                     → failed  ──→ 决策: 修复/追加/阻塞
                     → blocked ──→ 等待人工决策
```

## 门禁判定规则

| 门禁结果 | 条件 | 动作 |
|----------|------|------|
| `pass` | 无阻断项 | 任务标记 passed，推进下一任务 |
| `fail` | 存在阻断项 | 决策引擎判断路径：修复当前任务 / 追加修复子任务 / 标记 blocked |
| `fail` (第 N 次) | 同一任务连续 fail | 不触发熔断；追加新任务要求人工介入 |
| 熔断 | 总迭代轮数达上限 | 强制终止，标记所有未完成任务为 blocked |

## 示例对话

> **User:** "给项目加一个缓存层"
>
> **自动迭代:** "收到。先探查项目状态……项目是 Python + Redis 可用。目标已明确。"
> "拆解任务链：1) 缓存抽象层  2) 查询结果缓存装饰器  3) 写入失效策略  4) 缓存监控指标。开始执行。"
>
> **[迭代 1]** "执行任务 1：缓存抽象层…门禁 pass ✅"
> **[迭代 2]** "执行任务 2：查询缓存装饰器…门禁 fail ❌，blockers: [线程不安全, 缓存键冲突]。"
> "决策：任务 2 回退到 in_progress，追加子任务 2a（修复线程安全）和 2b（修复缓存键策略）。"
> **[迭代 3]** "执行任务 2a：修复线程安全…门禁 pass ✅"
> **[迭代 4]** "执行任务 2b：修复缓存键策略…门禁 pass ✅"
> **[迭代 5]** "重新执行任务 2：查询缓存装饰器…门禁 pass ✅"
> **[迭代 6-7]** "任务 3、4 依次通过。"
> **[收尾]** "任务链全部通过。4 个原始任务 + 2 个追加子任务，7 轮迭代，0 阻塞。日志: LOOP_LOG.md。"
