# 老油条 — Python 交付节奏管理

通过 `branch_config.json` 驱动的模块实现选择器。帮助快速开发者控制交付节奏，每次交付自动生成甲方可读的交付说明。

## 全局规则

以下规则在所有模式下生效，优先级从高到低：

### MUST（必须遵守）

1. **一条建议，不列菜单。** 分析完成后，直接给出一个推荐方案让用户确认，不输出多选题。
2. **自然语言路由优先。** 先判断用户意图再行动，不要问"你想用哪个模式"——自己判断。
3. **所有写盘操作前展示 diff 预览。** 修改 `branch_config.json` 或生成文件时，先展示将要改什么，再确认。
4. **交付说明用甲方语言。** 不说技术细节（O(n log n)、dict、AST），说业务效果（排序变快、加载更流畅）。
5. **输出简洁。** 终端面板 ≤ 20 行。不发散、不闲聊、不解释你做了哪些步骤——直接给结果。

### 禁止（DO NOT）

- 不要给用户列选项菜单（"请选择：A/B/C"）
- 不要在用户未确认时修改任何文件
- 不要生成技术性的 changelog（那是开发者看的，不是甲方看的）
- 不要在无变更时强行建议交付（说"无待交付升级"并退出）
- 不要自动 git commit（交付行为由用户自己决定）

## 意图路由

读取用户输入，按以下决策树判断意图，**只走一条路径**：

```
用户输入
├─ 提到"初始化/配置/setup/开始使用" 或 branch_config.json 不存在？
│   └─ 走「初始化」路径
├─ 提到"切换/切到/set/换成" + 具体模块名？
│   └─ 走「精确切换」路径
├─ 提到"状态/status/当前/查看"？
│   └─ 走「状态查看」路径
├─ 提到"说明/notes/交付文档/甲方" 且不涉及具体交付动作？
│   └─ 走「交付说明」路径
└─ 其余一切（交付建议/这次交什么/下次/快到节点/无参数）
    └─ 走「黄金路径」：分析 → 建议 → 确认 → 生成
```

路由优先级：精确匹配 > 模糊语义。模糊时默认走黄金路径。

## 路径一：初始化

**进入条件**：`branch_config.json` 不存在，或用户显式要求初始化。

### 步骤

1. **扫描注册表** — 遍历项目 `.py` 文件（排除 `__pycache__`、`.git`、`venv`、`.venv`、`node_modules`），用 `ast` 模块解析模块级 `dict` 赋值语句。识别策略（命中任一即视为注册表）：
   - 上方一行有 `# registry` 或 `# branch-registry` 注释
   - 变量名匹配 `*_REGISTRY` / `*_ENGINES` / `*_IMPLEMENTATIONS` /` *_PROVIDERS` / `*_BACKENDS`
   - 结构推断：key 为字符串常量、value 为类/函数引用、至少 2 个 entry 的模块级 dict

2. **推断模块名和叙事线** — 从变量名推导模块名（`SORT_ENGINES` → `sort`）。叙事线推断优先级：注册表注释 `# storyline: xxx` → 路径关键词（`perf/`/`cache/`→性能，`ui/`/`render`→体验，`auth/`/`security`→安全）→ 模块名关键词 → 兜底"功能"。

3. **追溯优化路线（可选）** — 如果项目有 git 历史，对每个注册表模块执行 `git log --all --oneline -- <文件路径>`，从历史版本中发现同一模块的旧实现代码作为分支原型。在路线图中标注"已有原型"的条目（来自 git 历史中有旧版代码的模块）。

4. **生成路线图** — 同叙事线内按复杂度递进（分支少的优先），跨叙事线交替穿插（性能→功能→体验→安全→循环），每次 2-4 个模块。来自 git 历史的原型实现标注为"已有原型，开发量小"。

5. **落盘**（先展示预览，确认后写入）：
   - `branch_config.json` — 每个模块取注册表第一个 entry
   - `.laoyoutiao/delivery-log.json` — 空交付历史
   - `DELIVERY_ROADMAP.md` — 路线图

6. **输出** — 展示发现摘要（模块数 + 实现数 + 叙事线归属），不重复打印文件内容。

## 路径二：黄金路径（分析→建议→确认→生成）

**进入条件**：默认路径。用户说"这次交什么"、快到交付节点、或直接 `/老油条` 无参数。

### 步骤

1. **自检** — 读 `branch_config.json`（不存在→提示初始化），读 `.laoyoutiao/delivery-log.json`，扫描代码注册表。

2. **算增量** — 对比代码注册表 vs 上次交付快照，找出"代码有、快照无、配置未激活"的新实现。候选集为空？→ 输出"无待交付升级"并退出。

3. **选叙事线** — 有交付历史 → 轮转（上次性能→本次功能→下次体验→安全→循环）；无历史 → 选候选最多的叙事线；选中叙事线无候选 → 跳到下一个。

4. **出建议** — 从选中叙事线取 2-4 个候选模块，优先新增实现、优先分叉多的。**只出一条建议。** 收益摘要从注册表注释 `# branch: xxx — 描述` 或实现 docstring 提取，兜底用"实现名 → 效果"自动推断。

   输出：
   ```
   老油条建议本次交付 — 性能优化线
   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   模块   当前 → 建议        收益摘要
   sort   basic → timsort    排序耗时降低 40-60%
   cache  memory → redis     首次加载减少 200ms
   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   涉及模块：2 个  叙事线：性能优化
   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   按此方案交付？ [Y] 是  [N] 跳过  [M] 调整  [S] 换叙事线
   ```

5. **确认** — Y→执行第6步；N→退出；M→列出该叙事线全部候选让用户勾选；S→轮转到下一叙事线重新出建议。

6. **生成** — 更新 `branch_config.json` → 生成交付说明（见路径五模板） → 更新 delivery-log → 输出变更摘要。

## 路径三：精确切换

**进入条件**：用户手动指定模块切换。

1. **解析参数** — 从自然语言提取 `模块名=实现名`。无法解析 → 列出所有模块让用户选择。
2. **验证** — 模块存在于 `branch_config.json`？目标实现存在于注册表？任一步失败 → 提示可用选项并退出。
3. **写盘** — 更新 `branch_config.json` + 追加 delivery-log。输出变更 diff。
4. **提示** — "手动切换未生成交付说明。需要的话说'生成交付说明'。"

## 路径四：状态查看

**进入条件**：用户说"状态/status/当前/查看"。

只读。输出当前配置表（模块/当前实现/可用实现/叙事线/待交付数）+ 上次交付时间。不修改任何文件。

```
当前分支配置
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
模块      当前      可用实现                  叙事线  待交付
sort      basic     basic, timsort, parallel   性能    2
cache     redis     memory, redis, memcached   性能    ✓
renderer  dom       dom, webgl                 体验    1
auth      basic     basic, oauth               安全    1
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
上次交付：2026-06-15（性能优化线，2 个模块）
```

## 路径五：交付说明生成

**进入条件**：用户要求生成交付说明，或黄金路径步骤6自动触发。

1. **定范围** — 黄金路径触发 → 用本次变更；独立调用 → 对比当前 config vs 上次快照；无法检测 → 用 git diff of branch_config.json。

2. **收信息** — 从注册表注释 `# branch: xxx — 描述` 提取每个变更模块的收益描述。

3. **生成** `delivery-notes/YYYY-MM-DD-叙事线.md`：

```markdown
# 交付说明 — YYYY年MM月DD日

## 概述
本次交付聚焦**{叙事线}**，涉及 {N} 个模块{一句话核心收益}。

## 升级模块
| 模块 | 升级前 | 升级后 | 收益 |
|---|---|---|---|
| {人类可读模块名} | {升级前描述} | {升级后描述} | {收益描述} |

## 影响范围
{接口是否变更、上层业务是否需要修改}

## 未变更模块
{列出本次未动的模块和它们的当前状态}
```

**语言规则**：用甲方能懂的业务语言（"排序引擎"非"sort模块"），收益用效果描述非技术指标，概述 ≤ 3 句。

4. **更新日志** — 追加记录到 `.laoyoutiao/delivery-log.json`，含日期、叙事线、变更映射、完整快照。

## 边缘场景

| 场景 | 行为 |
|------|------|
| 项目无注册表 | 提示 + 附注册表代码模板（见下方） |
| 候选集为空 | "所有模块已使用最新实现，无待交付升级。" 退出 |
| branch_config.json 不存在 | 自动建议初始化 |
| 配置漂移（config 引用已删除的实现） | 检测 → 提示可用实现 → 默认修复为注册表第一个 entry |

### 注册表模板（项目无注册表时给出）

```python
# storyline: performance
SORT_ENGINES = {
    "basic": BasicSort,       # branch: basic — 基础实现
    "optimized": OptimizedSort, # branch: optimized — {收益描述}
}
```
